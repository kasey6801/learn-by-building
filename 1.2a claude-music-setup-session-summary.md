# How This Was Built — A Session Summary

## Project overview

**Claude + Apple Music Setup** is a one-file macOS installer (`Claude Music Setup.command`) that connects Claude Desktop to the user's Apple Music library via a music MCP connector. Once installed, the user can ask Claude to build historically-informed playlists in plain English — sampling lineage, genre origins, compositional influence — and Claude will create them directly in Apple Music.

**v1 scope:** Apple Silicon Macs only. Requires Claude Desktop and an Apple Music subscription. Connects via `@pedrocid/music-mcp@latest` — fetched on demand by npx each time Claude starts a music session. The connector works only with songs already in the user's personal library. Delivered as a `.dmg` containing the installer and a README. Five-step automated flow with a single manual permission step.

**Future iterations may include:** Intel Mac support, a code-signing certificate to eliminate the Gatekeeper warning, an uninstaller option, and a companion onboarding guide for first-time Apple Music library prompts.

This project serves a dual purpose. The first is practical — a genuinely useful tool that requires no technical knowledge to install or use, and that unlocks a category of music discovery conversation that is not possible in Claude Desktop out of the box. The second is a learning demonstration: showing how a non-developer can work with AI assistance to produce polished, distributable developer tooling — a shell script, a JSON config merge, a macOS `.dmg` — without writing a line of code manually.

## Overview

This document records the working session in which the Claude Music Setup installer was designed and written. The installer was produced through a structured conversation: a non-developer working as the product owner, and Claude acting as a combination of technical architect, script author, and documentation writer.

The session began with a simple goal — connect Claude Desktop to Apple Music — and moved through a series of decisions that progressively shaped the installer's scope, delivery format, error handling, and UX. By the end, the output was a complete, working installer and a full README, ready to package and distribute.

---

## Prompt 1

*"I'd like to be able to ask Claude to build Apple Music playlists for me — like 'build a playlist of the most sampled 70s funk tracks.' Is there a way to connect Claude Desktop to my Apple Music library?"*

Claude identified the MCP connector ecosystem as the right approach and located `music-mcp` by Pedro Cid (`@pedrocid`) as the only mature, MIT-licensed connector for Apple Music on macOS. The response explained how MCP connectors work in plain terms — small background processes that give Claude a set of tools it can call — and confirmed that `music-mcp` uses AppleScript to communicate with the Music app, meaning it operates entirely on the user's Mac with no third-party service involved. The core architecture was established: Claude Desktop → MCP protocol → music-mcp connector → AppleScript → Music app.

**Asset created:** None — the output was a research and recommendation response. The value here is the decision it produced: `music-mcp` via npx rather than a custom-built connector. Using an existing, maintained, open-source package meant the project could focus entirely on the installer experience rather than building and maintaining connector code.

---

## Prompt 2

*"I want this to be easy for non-technical people to set up. What's the simplest way to deliver it — something people can just double-click?"*

Claude proposed the `.command` + `.dmg` combination as the optimal delivery format for macOS. A `.command` file is a bash script that macOS opens in Terminal when double-clicked — requiring no Homebrew, no developer tools, no command line knowledge from the user. A `.dmg` disk image is the standard macOS distribution format: familiar, self-contained, and easy to attach to a GitHub release or share as a download link. The alternative (a proper `.pkg` installer with a GUI wizard) was noted as higher effort with no meaningful UX advantage given the simplicity of the setup steps.

**Asset created:** Delivery format decision — `.command` script inside a `.dmg`. This shaped every subsequent decision in the project: the installer needed to be a self-contained script that read progress from the terminal, with clear step-by-step output, rather than a GUI application. The choice to use Terminal output rather than a GUI installer turned out to be a strength: the colour-coded step output is more transparent and informative than a progress bar would be.

---

## Prompt 3

*"What should the installer actually do? Walk me through what it needs to handle."*

Claude produced the five-step flow that became the installer's structure:
1. Confirm macOS and Apple Silicon (AppleScript is macOS-only; Intel out of scope for v1)
2. Confirm Claude Desktop is installed (nothing to configure without it)
3. Install Node.js if absent or outdated (npx requires Node.js 18+)
4. Add the music server entry to Claude Desktop's config file
5. Open the macOS Automation permission screen

Each step was explained with its rationale: why it is needed, what it checks, and what the failure path looks like. The decision to use `npx @pedrocid/music-mcp@latest` rather than a permanent global install was made here — on-demand npx means the connector is always current without any update step, and nothing is left installed on the system if the user later removes the config entry.

**Asset created:** Five-step installer flow (initial specification). The architectural backbone of the script. The rationale for each step informed the comments that now appear in the script — making the installer self-documenting.

---

## Prompt 4

*"For the config file step — how do we handle people who already have other connectors set up? We can't wipe their existing config."*

Claude identified three scenarios that the config step needed to handle: no file exists (first-time Claude user), file exists and is valid (most users), file exists but is malformed (edge case from manual editing). The recommendation was to use Python 3 — which ships pre-installed on macOS — rather than `sed` or `awk` for JSON manipulation, because Python's `json` module is the only safe way to guarantee the output remains valid JSON regardless of what is already in the file. The backup behaviour (timestamped copy of any malformed file before overwriting) was added as a recoverable failure path rather than a hard abort.

**Asset created:** Config merge logic specification — Python-based JSON read/modify/write with three-scenario handling and backup on malformed input. This decision is the most technically consequential in the project. A simpler text-based approach would have failed silently on edge cases, potentially breaking Claude Desktop's entire connector configuration. The Python approach handles all three scenarios robustly and leaves a recovery path in the worst case.

## Prompt 5

*"What about the terminal output — can we make it look clean and readable? Not just plain text."*

Claude specified the colour-coded output system using `tput` — the POSIX-standard tool for reading terminal capabilities. Four message types were defined, each with a distinct visual treatment: `step` (cyan bold, `▶` prefix) for major step headings; `success` (green, `✓`) for completed actions; `warn` (yellow, `⚠`) for non-fatal notes; `fail` (red, `✗`) for errors before exit. The `tput || echo ""` fallback pattern was specified to ensure clean output on terminals that do not support colour, with no stray escape characters.

**Asset created:** Output formatting system — four helper functions (`step`, `success`, `warn`, `fail`, `info`) with `tput` colour codes and fallback. This is a UX decision as much as a technical one: a non-technical user running the installer in Terminal needs to be able to see at a glance what is happening and whether each step succeeded. The visual hierarchy makes the five-step flow scannable even without reading every line.

## Prompt 6

*"Write the full installer script."*

Claude produced the complete `Claude Music Setup.command` bash script. The five-step flow was implemented with all specified logic: `uname` and `uname -m` checks, `pgrep` for the running Claude check, the `install_node()` function with `curl` download and `sudo installer`, the three-scenario Python config merge with backup, and the `x-apple.systempreferences:` deep-link for the Automation permission screen. Error handling was applied throughout using `fail()` calls with plain-language messages pointing users toward the resolution. The header block and inline comments were written to make the script fully readable to someone without bash experience.

**Asset created:** `Claude Music Setup.command` — complete working installer script. The primary deliverable of the project. The script is dense with comments because the intended audience includes users who want to review it before running it (the README explicitly encourages this), and because transparent, self-documenting code builds trust for a tool that requests a Mac password.

## Prompt 7

*"Write the README for the .dmg. It should be thorough — covering installation, how the connector works, the library limitation, and how to turn it off."*

Claude produced the complete README, structured in five sections: installation steps (with images), a technical explanation of how MCP connectors and music-mcp work, the library limitation and its practical implications, instructions for managing the connector (disable, re-enable, permanent removal), and troubleshooting. The library limitation section was written to be proactively honest rather than buried — it affects every use of the tool and needs to be understood upfront so users know to ask Claude for Apple Music links alongside playlists. Example prompts were included across three musical domains (sampling/hip hop, folk, classical) to demonstrate the range of queries the setup enables.

**Asset created:** `README.md` — complete user-facing documentation for the `.dmg`. The README serves the same role as a product's onboarding experience: it shapes the user's understanding before they encounter the limitation for the first time in a live session. Writing the library limitation clearly and constructively (here is what is missing, here is exactly how to work around it) prevents the most common source of confusion and frustration.

## Prompt 8

*"Add the completion message at the end of the script — a summary of what to do next."*

Claude added the closing block to the installer: a separator line, "Setup complete." in green, and a structured two-step post-install checklist (grant the permission, restart Claude Desktop). The closing block also included a sample prompt the user can paste directly into Claude Desktop as their first test, and the tip about requesting Apple Music URLs alongside playlists. The framing throughout was conversational and direct — instructions the user can follow immediately, not documentation to file away.

**Asset created:** Installer closing block — post-setup instructions and first-use prompt embedded directly in the terminal output. Ending the installer with a concrete next action (a copy-paste prompt) reduces the gap between "setup finished" and "I understand what this tool can do." The sample prompt is the same one used in the README, creating a consistent first impression across both surfaces.

## Prompt 9

*"Review the script for anything that could go wrong from a non-technical user's perspective — error cases we haven't handled."*

Claude reviewed the script and identified two gaps. First, the Node.js version check used `(( NODE_VERSION >= NODE_MIN_VERSION ))` arithmetic comparison, which would fail silently if the `node --version` output contained unexpected formatting. The `sed 's/v//'` and `cut -d. -f1` pipeline was confirmed correct and the check was left as written. Second, the PATH export after Node.js installation needed to be explicit (`export PATH="/usr/local/bin:/usr/bin:$PATH"`) rather than relying on the shell to inherit the updated path after the `sudo installer` call — this was added to prevent a failure mode where `npx` was not found immediately after installation. No user-facing changes were needed, but the PATH fix was confirmed important for reliability on clean Macs.

**Asset created:** PATH export fix — `export PATH="/usr/local/bin:/usr/bin:$PATH"` added immediately after the Node.js installation call. A single line, but a meaningful reliability improvement: without it, the script would fail on a clean Mac at the `node --version` verification step immediately after installing Node.js, because the shell's PATH has not yet been updated to include the new binary.

## Prompt 10

*"Package it — what goes in the .dmg and what do I call it?"*

Claude specified the `.dmg` contents (two files: `Claude Music Setup.command` and `README.md`), the disk image name (`claude-music-setup.dmg`), and the steps to create it using macOS Disk Utility (New Image from Folder). Notes were added on file permissions: the `.command` file needs execute permission (`chmod +x`) before packaging, since a `.dmg` created from a folder preserves the permissions of the source files. The Gatekeeper warning users would encounter on first run — because the file is not signed with a paid Apple Developer certificate — was noted, along with the correct user response (Done → Open Anyway in the privacy dialog) and the phrasing used in the README to explain it.

**Asset created:** Distribution specification — `.dmg` contents, naming, creation method, and permission requirements. The Gatekeeper handling turned out to be the most important UX consideration in distribution: without a clear explanation in the README, users encountering the "unidentified developer" warning would reasonably abandon the installation. The README addresses this explicitly with screenshots and step-by-step guidance.
